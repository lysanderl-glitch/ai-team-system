SPE 使用指引

总裁个人效率管理系统 v1.0

📅 2026-04-12 📦 Phase 1-4 全部交付 🏢 Janus Digital

SPE 是 Synapse 体系的个人任务管理模块，为总裁提供从任务捕获、日程规划、时间管理到每周回顾的完整闭环。四大捕获渠道 + 四个核心命令 + 自动化 Agent，让每一件事都不遗漏。

一、体系概览

Synapse Personal Engine (SPE) 是 Synapse AI 协作体系中专注于总裁个人效率管理的模块。它将分散在各处的待办、想法、日程、决策汇聚到统一入口，通过 AI 自动分流、智能规划、定时复盘，实现零遗漏、低打扰、高产出的个人管理闭环。

体系架构

SPE 采用四层架构，信息从捕获到回顾形成完整循环：

CLI / Web / Slack / Calendar — 四大渠道零遗漏收集

SPE 与 Synapse 的关系

Synapse 体系的核心是团队协作管理：44 位 Multi-Agent 成员在 Lysander CEO 统筹下完成各类任务。而 SPE 补全了体系中个人管理维度的缺失——总裁自身的待办、日程、决策、OKR 也需要系统化管理。

SPE 与 Synapse 主体系无缝集成：捕获的任务可分流为团队任务（通过 /dispatch 派单）或个人任务（保留在 SPE 中）；决策日志与 Synapse 四级决策体系共用；每周回顾自动关联团队产出数据。

三、自动化机制

SPE 的自动化层让系统在你不在线时也持续运转，减少手动操作，最大化效率。

3.1 SessionEnd Hook — 会话结束自动捕获

当你关闭 Claude Code 会话时，capture_session_tasks.py 脚本自动运行，从对话记录中提取未捕获的行动项，写入 personal_tasks.yaml 收件箱。

工作原理

  1. Claude Code 会话结束时触发 SessionEnd hook
  2. 脚本从 stdin 接收 JSON 输入（含对话记录路径和会话 ID）
  3. 读取 JSONL 格式的对话记录
  4. 用关键词匹配提取候选行动项（中文/英文关键词 + checkbox 模式）
  5. 与已有条目去重后，新项追加到 inbox
  6. 自动 git commit 保存变更

匹配的关键词

语言关键词
中文需要、记一下、回头、待办、后续、跟进、别忘了、提醒我
英文TODO、action item、follow up、remind me、need to
特殊- [ ] 开头的 checkbox 行

配置方法

将以下配置加入 Claude Code 的 settings.json（全局 ~/.claude/settings.json 或项目级 <project>/.claude/settings.json）：

{ "hooks": { "SessionEnd": [ { "matcher": "", "hooks": [ { "type": "command", "command": "PYTHONUTF8=1 python3 \"/c/Users/lysanderl_janusd/Claude Code/ai-team-system/scripts/capture_session_tasks.py\"", "timeout": 30000 } ] } ] } }
⚠️ 注意

如果 settings.json 中已有其他 hooks 配置，将 SessionEnd 数组合并到现有 hooks 对象中，不要覆盖已有的 hook 配置。此配置只需执行一次。

六、智能化特性

🧠
行为观察
AI 持续学习你的工作模式，越用越懂你

SPE 的智能化层会在后台持续观察四个维度：

⏱️
工作节奏
活跃时段、精力高峰、工作模式切换
🎯
决策模式
决策速度、风险偏好、信任度校准
📋
任务偏好
亲自做 vs 委派的类型分布
💬
沟通偏好
信息密度、过程参与度、关注焦点

这些观察数据以渐进方式积累，不会一次性要求你填写问卷。当系统发现可操作的模式时，会在 /plan-day 中自动体现（如调整建议时段、调整优先级排序）。

🔒 隐私保护

所有行为观察数据仅存储在本地 memory 系统中，不会上传到任何外部服务。配置详见 agent-butler/config/spe_intelligence.yaml。

📈
OKR 自动追踪
每日检查进度，及时预警落后项

每次执行 /plan-day 时，系统自动检查 OKR 进度：

  • 进度落后 → 在日程概览中标注 ⚠️ 警告
  • 7 天未更新 → 提醒确认该 KR 是否仍在进行
  • 目标无关联任务 → 建议分解为具体行动项

OKR 设定方式：告诉 Lysander 你的季度目标，他会更新 personal_tasks.yaml 的 personal_okrs 节。

📊
决策质量分析
30 天回顾 + 质量评分，持续提升决策水平

每条记录的决策在 30 天后自动触发回顾提醒，评估内容包括：

  • 预期结果是否实现？（1-5 分）
  • 是否有未预见的副作用？
  • 如果重来会做不同选择吗？
  • 获得了什么经验教训？

系统还会跟踪决策准确率、领域分布和偏差模式，帮助识别哪些领域的决策需要改进。

七、常见问题

▶ Skill 不可用怎么办？
重新打开一个新的 Claude Code 会话即可。Skill（如 /capture、/plan-day 等）需要在新会话中才能加载。已有会话中首次使用的 Skill 会在当前会话中持续可用。
▶ Google Calendar 认证过期怎么办？
/plan-day 内置降级策略：当日历 API 不可用时，会跳过日历数据，仅基于 personal_tasks.yaml 和 Slack 数据生成规划，并提示你手动告知当天的重要日程。日历认证恢复后自动恢复完整功能。
▶ 如何修改工作时间范围？
编辑 agent-butler/config/calendar_config.yaml 的 work_hours 字段。例如将工作时间调整为 09:00-18:00：

work_hours:
  start: "09:00"
  end: "18:00"

同时建议相应调整 energy_rules 中的各时段划分。
▶ inbox 太多怎么处理？
执行 /plan-day 时，AI 会自动将 inbox 中的项目分流到四个类别：

→ 决策：需要你做判断的事项，进入决策队列
→ 委派：可以交给团队的任务，建议通过 /dispatch 派单
→ 个人：需要你亲自执行的事项，进入 Big Rocks 候选
→ 参考：仅供参考的信息，归档到 OBS 知识库

分流后 inbox 会自动清空。如果积累过多，直接运行一次 /plan-day 即可批量处理。
▶ 如何设置 OKR？
直接告诉 Lysander 你的季度目标和关键结果即可。例如：

"Lysander，我的 Q2 目标是：1. 建立个人品牌影响力——每周发布 1 篇博客、播客访谈笔记沉淀 10 篇；2. 完成 Janus 项目交付——客户满意度 4.5 分以上。"

Lysander 会将其结构化录入 personal_tasks.yaml 的 personal_okrs 节，此后每次 /plan-day 都会自动追踪进度。
▶ SPE 数据存在哪里？安全吗？
所有 SPE 数据存储在本地文件系统（agent-butler/config/ 目录下的 YAML 文件）和本地 Git 仓库中。行为观察数据存储在 Claude Code 的 memory 系统中。没有任何数据会上传到第三方服务。Google Calendar 和 Slack 的集成通过 MCP 实时读取，不缓存外部数据。
▶ 时间块的时段是如何自动选择的？
SPE 基于 calendar_config.yaml 中的精力分配规则匹配最佳时段：

08:00-12:00（高精力）→ 深度工作、关键决策
13:00-15:00（中精力）→ 会议、协作
15:00-18:00（常规）→ M 级任务
18:00-20:00（低精力）→ S 级轻量任务、复盘

同时会检查 Google Calendar 上已有事件，避免时间冲突。如果指定时段已满，会自动选择下一个可用的同类时段。
