方法论

Harness Engineering
2026年AI工程新范式 — Agent = Model + Harness

Guides — 前馈控制
CLAUDE.md → 角色、规则、流程
organization.yaml → 团队架构、路由
decision_rules.yaml → 决策约束
HR/personnel/*.md → 能力档案
Sensors — 反馈控制
QA 自动评分 → 质量门禁
执行链检查 → 流程完整性
决策日志 → 误判追踪
专家评审团 → 多维度评分
竞争优势不在于谁的模型最大，
而在于谁的 Harness 最有效。
— Martin Fowler / Red Hat, 2026
场景三 · 自动化

每日情报 → 行动 全自动闭环

6:00am ─── 任务恢复Agent ─── 检查未完成工作，自动续接 │ 8:00am ─── 情报日报Agent ─── 搜索AI前沿 → 筛选 → 分析报告 │ → HTML → git push → Slack通知 │ 10:00am ── 行动管线Agent ─── 提取建议 → 4专家评估(1-5分) │ → 评审决策 → Harness Ops执行 │ → 成果报告 → Slack通知 │ 每周一 ─── HR审计Agent ──── 全员质量审计 → 自动修复 → 报告

总裁只在 Slack 上收到最终通知。整个体系无需人工启动即持续运转。

场景五 · 发现问题并修复

从"口头承诺"到 制度强制

发现的问题
Lysander 反复跳过团队派单环节
自己一个人做完所有工作
执行链写了"路由到团队"
但没有机制检查是否真的路由了

根因：有 Guides，没 Sensors
建立的制度
强制团队派单制度
M/L级任务执行前必须输出派单表
每个工作块标注执行者
与开场问候同级，不可省略

Guide + Sensor 双保险

体系不靠"下次注意"，靠制度强制。

场景六 · 组织质量

HR 管理体系 — Agent 也需要人力资源

问题：Agent 想加就加，加完没人管，管了没标准。
审计发现：Janus 6人全部 0 分，Growth 全员 0 分。

方案：建立完整的 Agent 人力资源管理体系 —

• 强制 Schema — 每个 Agent 必须有完整档案
• 能力质量分级 — A/B/C 三级，C 级不合格
• 入职审批 — HR Director 8 项检查清单
• 周度自动审计 — 每周一自动评分全员
• 情报联动进化 — 新技术出现 → 自动更新能力
• 退役制度 — 不达标就降级或退役
