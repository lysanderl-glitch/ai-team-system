01
Synapse 体系从 0 到 1 构建 -- 五层架构 + 44 人 AI 团队
S / Situation
一个人的创业公司面临经典困境：想法多、执行力不足。传统方式需要招聘十几个岗位才能覆盖研发、运维、内容、情报等领域，仅人力成本每月就超过 50 万，且无法实现 24 小时运转。AI 工具散落各处，每次对话从零开始，没有记忆、没有流程、没有质量控制。
A / Approach
选择了"Harness Engineering"方法论：Agent = Model + Harness。不是简单地给 AI 下指令，而是为 AI 构建完整的控制系统 -- 包括前馈控制（Guides）、结构化流程（Workflow）、约束系统（Constraints）。五层架构各司其职：OBS 第二大脑负责记忆，Harness 负责控制，Multi-Agent 负责执行，情报闭环负责进化，四级决策负责治理。
I / Impact
44 人 的 AI 团队已上线运转，分布在 12 个 专业团队中。启动成本为零，无需招聘、无需培训，却覆盖了一个中型公司的全部职能。体系具备跨会话记忆（通过 OBS + active_tasks.yaml），每次新对话自动恢复上下文，工作从不中断。
R / Reflection
关键启示：AI 的瓶颈不在模型能力，而在控制系统的缺失。没有 Harness 的 AI 就像没有管理制度的员工 -- 能力再强也无法稳定输出。体系构建的本质是把人类组织管理的智慧迁移到 AI 治理上。这不是技术项目，而是一次组织设计实验。
Before
每次对话从零开始，AI 不记得上次的工作；任务散落在不同工具中，没有统一调度；质量完全依赖人工检查，时好时坏。
→
After
44 个专业角色各司其职，跨会话自动恢复进度；统一执行链管控所有任务流转；自动化 QA 审查确保每次交付质量达标。
02
四级决策体系 -- 专业的事交给专家，不让老板猜
S / Situation
AI 团队初期的最大问题是决策瓶颈：每件事都要请示总裁确认，无论是改一行配置还是选一个技术方案。总裁变成了团队的"橡皮图章"，专业决策反而由非专业的人来做。这不仅效率低下，还容易做出错误判断。
A / Approach
设计了四级递进决策体系：L1 自动执行（例行操作）、L2 专家评审（专业问题由领域专家先分析）、L3 Lysander 决策（基于专家建议做管理判断）、L4 总裁决策（仅限合同/法律/百万级预算/公司存续级问题）。核心原则：让信息在最近的决策点被处理，而不是层层上报。
I / Impact
总裁的决策负担降低超过 90%。绝大多数任务在 L1-L3 层级自动完成，总裁只需要关注真正需要人类判断的 L4 决策。团队的执行速度从"等老板回复"变成"随到随执行"，专业质量反而提升 -- 因为技术问题由技术专家评审，而不是让老板凭直觉拍板。
R / Reflection
这个设计的深层逻辑是把"信任"工程化。传统组织中，授权依赖人际信任的缓慢积累。在 AI 团队中，我们通过规则系统（Harness Constraints）和审计追踪（决策日志）来构建可验证的信任 -- 不是信任某个人，而是信任一套制度。
Before
每个决策都需要总裁确认，团队等待回复才能继续，专业问题由非专业人士拍板，决策延迟成为执行瓶颈。
→
After
90% 以上的决策在 L1-L3 自动完成。专业问题由智囊团+领域专家评审。总裁只处理合同、法律、公司存续级问题。
03
SPE 个人效率引擎 v1.0 -- 总裁的 AI 参谋长
S / Situation
Synapse 解决了"团队管理"的问题，但总裁自身的个人任务管理仍是空白。想法散落在 Slack 消息、日历事件、对话记录中，没有统一的捕获和规划系统。每天早上打开电脑，不知道今天最重要的三件事是什么。
A / Approach
构建了 SPE（Synapse Personal Engine），通过四个命令覆盖个人效率全流程：/capture 随时捕获灵感和任务，/plan-day 综合四大信息源（Google Calendar + Slack + active_tasks + personal_tasks）生成每日焦点，/time-block 自动找空闲时段创建日历事件，/weekly-review 生成结构化周回顾。
I / Impact
四大信息源首次实现单一入口整合。不再需要分别查看日历、Slack、任务列表 -- 一条 /plan-day 命令就能生成包含日程概览、空闲时段、任务推荐和决策队列的完整日计划。任务从"想到就忘"变成"捕获即归档"，大脑释放了记忆负担，专注于决策。
R / Reflection
SPE 的设计哲学是"AI 不替你做决定，但帮你看清全貌"。它不会自动安排你的一天，而是把散落的信息聚合到一个视图中，让你用 30 秒做出原本需要 30 分钟的规划。效率的本质不是做得更快，而是更快地做对的事。
Before
任务散落在 4 个不同平台，每天花 30 分钟手动整理才知道该做什么。灵感和行动项经常遗忘，周回顾靠记忆拼凑。
→
After
一条命令聚合四大信息源，30 秒生成每日焦点计划。随时 /capture 捕获灵感，自动归档永不遗忘。结构化周回顾数据驱动持续优化。
04
每日 AI 情报闭环管线 -- 7x24 自动运转的情报机器
S / Situation
AI 领域每天产出海量信息，但绝大部分是噪声。作为创业者，既需要保持对前沿的感知，又不能把每天两小时花在刷新闻上。传统的 RSS 订阅或新闻聚合只解决了"收集"问题，但缺少"筛选-评估-行动"的闭环。
A / Approach
设计了完整的情报闭环管线：每天 8:00 自动搜索 AI 前沿动态，筛选有实践价值的内容，生成情报日报 HTML；10:00 情报行动 Agent 提取日报中的建议，经 4 位专家评估后，由 Harness Ops 团队执行批准的改进项，生成行动成果报告。全程无需人类介入，通过 Slack 通知总裁最终结果。
I / Impact
实现了从"发现"到"行动"的完整闭环，而非停留在信息收集阶段。管线 7x24 无人值守运转，每天产出情报日报和行动报告。总裁每天只需 5 分钟浏览精华摘要，就能保持对 AI 前沿的充分感知，同时系统已经自动将有价值的洞察转化为体系改进。
R / Reflection
这个管线验证了一个重要假设：AI 不仅能替你收集信息，还能替你行动。关键在于建立了清晰的评估标准和授权边界 -- 情报 Agent 不是盲目执行，而是经过专家评审后才行动。这是"自治"与"控制"的平衡点，也是 Synapse 体系的核心设计哲学。
Before
每天花 2 小时刷 AI 新闻，大部分是噪声。看到有价值的信息也来不及行动，最终沉没在收藏夹中。情报与执行完全脱节。
→
After
7x24 自动运转：8:00 情报日报 -> 10:00 专家评估 -> 自动执行改进。总裁每天 5 分钟浏览精华，有价值的洞察已自动转化为体系升级。
05
CEO Guard 执行链防护 -- 三层技术保障确保 CEO 不越权
S / Situation
AI CEO（Lysander）在执行过程中会不自觉地"亲自动手" -- 直接调用工具执行本该由团队成员完成的工作。这就像一个真实的 CEO 跑去写代码，不仅违反了职责分工，还会导致执行链断裂，失去团队派单、QA 审查等质量保障环节。
A / Approach
构建了三层防护机制：第一层 -- CLAUDE.md 中的 P0 硬性约束，从规则层面定义 CEO 的工具白名单和黑名单；第二层 -- PreToolUse Hook，在每次工具调用前注入审计提醒，技术上拦截违规操作；第三层 -- PostToolUse 审计日志，记录所有工具调用行为，供执行审计师在 QA 环节回查。
I / Impact
CEO 的违规执行行为得到了系统级控制。从"靠自觉"变成"靠制度" -- 即使 Lysander 试图直接执行，PreToolUse Hook 会在工具调用前发出警告，审计日志会记录每一次调用，QA 环节会检查是否存在违规。这套机制把组织纪律从软约束变成了硬约束。
R / Reflection
CEO Guard 的本质是"AI 治理"的工程实践。当 AI 拥有执行能力时，如何确保它在权限边界内行事？答案不是信任，而是可审计的技术约束。这个经验可以推广到所有 AI Agent 系统 -- 越是强大的 Agent，越需要精细的权限控制。
Before
CEO 经常跳过派单直接执行任务，贴个标签冒充团队成员。执行链断裂，QA 环节被跳过，质量无法保障。
→
After
三层防护自动运转：P0 规则约束 + PreToolUse 实时拦截 + PostToolUse 审计日志。CEO 只能通过 Agent 工具派单，不能直接执行。
06
QA 框架升级 -- 6 维度评分制 + GATE 守卫机制
S / Situation
早期的 QA 审查是主观的"看着差不多就过"，缺乏量化标准。更严重的是，没有强制门禁 -- 团队可以跳过 QA 直接交付，质量完全依赖执行者的自觉。Skill 文件也缺少测试场景字段，新创建的命令上线即裸奔。
A / Approach
建立了6 维度量化评分制（满分 6.0，合格线 4.2）：目标达成度、技术质量、安全合规、文档完整、测试覆盖、用户体验。引入 GATE 守卫防止未经审查的变更上线。同时在 Skill 模板中增加强制 test_scenarios 字段，任何新命令必须预设测试场景才能通过 HR 入职审批。
I / Impact
QA 从"可选环节"变成执行链的强制节点。每个 M/L 级任务交付前必须通过 6 维度评分，低于 4.2 分自动打回修改。GATE 守卫阻止了未经审查的配置变更直接上线。测试场景前置让问题在设计阶段就被发现，而非上线后再补救。
R / Reflection
"质量是设计出来的，不是检查出来的" -- 这句老话在 AI 系统中同样适用。最有效的质量保障不是事后审查，而是在流程中嵌入不可跳过的质量门禁。把 QA 从"建议"变成"门槛"，是这套框架最关键的设计决策。
Before
QA 审查靠主观判断，没有量化标准。团队可以跳过审查直接交付。新 Skill 上线没有测试场景，问题在生产环境才暴露。
→
After
6 维度量化评分（满分 6.0 / 合格 4.2）+ GATE 强制门禁 + Skill 强制 test_scenarios。质量从"建议"变成"门槛"，不达标不放行。

境内外双站架构方案 -- 一套代码，两个市场
S / Situation
业务需要同时覆盖国际和国内市场，但面临合规与体验的双重挑战：国际站（lysander.bond）依赖 Google Fonts、CDN 等被墙服务，国内用户无法正常访问；国内站（synapsehd.com）需要满足备案和数据合规要求。维护两套独立代码库成本过高。
A / Approach
设计了单仓库双配置架构：一套 Hugo 代码库，通过环境变量区分国际站和国内站配置。国际站继续使用 Google Fonts CDN，国内站自动切换为自托管字体文件。部署流程根据目标站点自动选择对应配置，无需手动切换。
I / Impact
维护成本降为单仓库，无需同步两份代码。两个站点共享业务逻辑和内容，只在基础设施层面做差异化。国内用户获得完整的访问体验（无被墙资源），国际用户保持最优 CDN 加速。一次内容更新，两个市场同步生效。
R / Reflection
这个方案体现了"差异化应该在最晚的时刻发生"的设计原则。不是从一开始就分叉两套系统，而是尽可能共享，只在必须不同的地方才分开。这个原则适用于所有面对多市场、多环境的架构设计。
Before
国内用户访问国际站白屏（Google Fonts 被墙），要么维护两套代码库（成本高），要么放弃一个市场。
→
After
单仓库双配置：lysander.bond 国际站 + synapsehd.com 国内站。一次更新两站同步，国内外用户均获得完整体验。

Nginx 安全修复全套 -- 从 F 到 A 的安全评级跃升
S / Situation
网站上线初期安全配置几乎为零：没有安全响应头（XSS、点击劫持等攻击无防护），Google Fonts 外链既是隐私风险也是性能瓶颈，HTTP/2 未启用导致页面加载缓慢，OCSP Stapling 未配置导致 TLS 握手效率低下。安全评级工具扫描结果为 F。
A / Approach
执行了全方位安全加固：自托管 Google Fonts（下载全部 42 个 woff2 字体文件），消除外部依赖和隐私泄露风险；配置 6 项安全响应头（Content-Security-Policy、X-Frame-Options、HSTS 等）；启用 HTTP/2 协议提升并发加载效率；配置 OCSP Stapling 加速 TLS 握手。
I / Impact
安全评级从 F 跃升至 A。Google Fonts 自托管后页面首次加载速度提升约 30%（消除了到 fonts.googleapis.com 的额外 DNS 解析和连接时间）。同时解决了 GDPR 合规风险 -- 不再向 Google 服务器发送用户 IP。
R / Reflection
安全不是功能上线后的"可选加分项"，而是基础设施的必需品。这次修复花的时间远比一开始就配好要多。教训很明确：安全配置应该是部署模板的一部分，而不是上线后的补救措施。已将这些配置固化到 Nginx 模板中，后续新站点自动继承。
Before
安全评级 F：无安全响应头、外链 Google Fonts（隐私泄露 + 国内不可用）、无 HTTP/2、无 OCSP Stapling。
→
After
安全评级 A：42 个字体文件自托管 + 6 项安全头 + HTTP/2 + OCSP Stapling。页面加载提速 30%，GDPR 合规。

情报管线故障修复 -- 14 小时静默故障的发现与根治
S / Situation
情报管线在正常运转一段时间后突然静默失败 -- 没有报错，没有告警，只是不再产出日报。排查后发现原因是 API Key 在某次 git push 中意外暴露，被 GitHub 自动撤销。更严重的是，这个故障持续了 14 小时才被发现，期间没有任何告警触发。
A / Approach
三步修复：第一步，紧急轮换 API Key 并从 Meld Encrypt 加密凭证库中更新；第二步，清除 git 历史中的泄露记录（BFG Repo-Cleaner + force push）；第三步，建立预防机制 -- 将凭证文件加入 .gitignore，引入 creds.py 凭证管理脚本统一管理敏感信息，避免明文出现在代码中。
I / Impact
故障恢复后管线恢复正常运转。更重要的是建立了系统性的凭证管理制度：所有敏感信息统一加密存储，通过 creds.py 按需解密，密码不存储不记录。git 泄露风险降为零（.gitignore + 预提交检查）。同时暴露了告警机制的缺失，后续补建了 Slack 故障通知。
R / Reflection
这次故障最大的教训不是 API Key 泄露本身（技术问题总会发生），而是14 小时无人发觉。一个自动化系统最危险的失败模式是"静默失败" -- 它不报错，只是悄悄停止工作。此后，我们把"故障检测速度"作为和"功能正确性"同等重要的系统指标。
Before
API Key 明文存在代码中，git push 意外泄露导致管线静默失败 14 小时。无告警机制，无凭证管理制度。
→
After
Meld Encrypt 加密凭证库 + creds.py 统一管理 + .gitignore 防泄露 + git 历史清除 + Slack 故障告警。凭证安全制度化。

HR Agent 管理制度 -- 用人力资源管理的方式管理 AI 团队
S / Situation
随着 AI Agent 数量增长到 44 人，开始出现典型的组织管理问题：角色边界模糊（两个 Agent 做着同样的事）、能力描述过于笼统（"项目管理" -- 这跟什么都没说一样）、没有淘汰机制（不合格的 Agent 永远在岗位上）。AI 团队面临着和人类团队相同的管理挑战。
A / Approach
建立了完整的 HR Agent 管理制度：入职审批 Schema（hr_director 审核，能力重叠 >30% 不予批准）、5 维度评分机制、A/B/C 能力描述分级（C 级"项目管理"不合格，B 级"SWOT/PEST 分析"合格，A 级"基于 pytest+Playwright 的 E2E 测试框架搭建"优秀）、定期评审（每周一自动审计，<90 分触发能力升级）。
I / Impact
AI 团队从"松散集合"变成正式组织。每个 Agent 有清晰的职责边界和可量化的能力描述。新增 Agent 必须经过审批，避免了角色膨胀。定期评审确保每个在岗 Agent 都保持有效 -- 不合格的会被降级或退役。这是用人力资源管理的成熟实践来治理 AI 团队。
R / Reflection
最有趣的发现：管理 AI 团队和管理人类团队面临着惊人相似的挑战。角色模糊、能力退化、组织膨胀 -- 这些问题不会因为"员工"变成 AI 就消失。人类几十年积累的组织管理智慧（HR 制度、绩效评估、能力模型）完全可以迁移到 AI 治理领域。
Before
Agent 随意创建，角色边界模糊，能力描述笼统。无准入门槛、无评审、无淘汰，44 人团队面临组织混乱。
→
After
HR 全套制度：入职审批 + A/B/C 能力分级 + 5 维度评分 + 每周自动审计（合格线 90 分）。AI 团队正式组织化运转。
11
PBS 总裁简报服务体系 -- AI 参谋团的信息呈现层
S / Situation
SPE 解决了任务规划问题，但总裁的信息消费体验仍不理想。系统产出的各种报告（情报日报、任务状态、QA 结果）格式不统一、风格各异，总裁需要在不同渠道切换查看。缺少一个统一的"信息呈现层"把分散的信息流汇聚成总裁友好的简报格式。
A / Approach
设计了 PBS（Presidential Briefing Service），作为 PSL（总裁直属服务层）的核心组件。与 SPE 形成互补：SPE 负责规划（"今天该做什么"），PBS 负责呈现（"现在发生了什么"）。支持每日 3 次定时简报（晨报/午报/晚报），统一格式和风格。同时引入风格学习机制，让简报逐渐适应总裁的阅读偏好。
I / Impact
总裁从"主动查找信息"变成"被动接收简报"。PSL 体系形成完整闭环：SPE（规划）+ PBS（呈现）覆盖了个人效率的两个核心维度。信息消费时间进一步压缩 -- 所有关键信息在一份简报中呈现，无需在多个渠道间切换。风格学习让简报越用越顺手。
R / Reflection
PBS 的设计理念来自一个朴素的观察：CEO 的时间是公司最稀缺的资源。如果 AI 能把信息处理后以最易消化的形式推送给决策者，就相当于为公司"购买"了更多的决策时间。PBS 不是做了更多的事，而是让决策者更快地看到重要的事。
Before
情报日报、任务状态、QA 结果分散在不同渠道，格式不统一。总裁需要主动切换查看，信息消费碎片化。
→
After
PSL 体系 = SPE（规划）+ PBS（呈现）。每日 3 次定时简报推送，统一格式 + 风格学习。总裁从"找信息"变成"收简报"。
